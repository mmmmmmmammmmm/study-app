# 【Section 9】データベース

## 1. 関係データベース (RDB: Relational Database)
データを「行（レコード）」と「列（フィールド/属性）」から構成される2次元の「表（テーブル）」の形式で管理します。現在のシステム開発において最も一般的なデータベースです。

## 2. 主キーと外部キー（制約）
表同士を関連付け、データの矛盾を防ぐための重要なルールです。
*   **主キー (Primary Key)**: 表の中の行を「一意に（重複なく）」特定するための列です。社員表における「社員番号」などが該当します。主キーには「重複する値」や「NULL（データが存在しない空の状態）」を入れることはできません（一意性制約・非NULL制約）。
*   **外部キー (Foreign Key)**: 別の表の「主キー」を参照している列です。例えば、「社員表」にある「部署コード」列は、「部署表」の主キーを参照する外部キーです。これにより、存在しない部署コードを社員に登録できなくするルールを「参照制約」と呼びます。

## 3. データベース設計と正規化
データの重複（冗長性）をなくし、データを追加・更新・削除した際に矛盾（アノマリ）が発生するのを防ぐために、表を適切な形に分割していくプロセスを「正規化」と呼びます。
*   **第1正規形**: 1つのセルに複数のデータが入っている「繰り返し項目」を排除した状態。
*   **第2正規形**: 複合主キーの一部にのみ依存している項目（部分関数従属）を別の表に切り出した状態。
*   **第3正規形**: 主キー以外の項目に依存している項目（推移的関数従属）を別の表に切り出した状態。

## 4. トランザクション処理 (ACID特性)
データベースに対する「意味のある一連の処理のまとまり（例：口座Aから引出し、口座Bへ入金する）」をトランザクションと呼びます。信頼性を保つため、以下の4つの特性（ACID）が求められます。
*   **Atomicity (原子性)**: 処理が「完全に実行されるか」「全く実行されなかったことになるか」のどちらかであること（All or Nothing）。中途半端な状態で終わることは許されません。
*   **Consistency (一貫性)**: トランザクションの前後で、データベースの制約（残高がマイナスにならない等）が常に保たれていること。
*   **Isolation (独立性 / 隔離性)**: 複数のトランザクションを同時並行で実行しても、お互いに干渉せず、順番に単独で実行した場合と同じ結果になること。これを実現するために「排他制御（ロック）」が行われます。
*   **Durability (耐久性 / 永続性)**: トランザクションが正常に完了（コミット）したら、その直後に停電やシステム障害が発生しても、結果は永久に失われないこと。ログファイルを用いて障害回復（ロールフォワード等）を行います。
